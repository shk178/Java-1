1. 기존 WAR 배포 방식의 주요 단점은 무엇일까요?

A
애플리케이션 코드 수정이 어려움

WAS(웹 애플리케이션 서버)를 별도로 설치하고 설정해야 함

C
HTTP 요청 처리가 느림

D
데이터베이스 연결 설정이 복잡함



해설
WAR 배포 방식은 톰캣 같은 WAS를 따로 설치하고 환경 설정을 해줘야 해서 과정이 복잡하고 번거로웠습니다.
2. 내장 웹 서버 개념의 핵심 아이디어는 무엇일까요?

A
웹 서버 기능을 사용하지 않음

B
외부 서버에 종속적

웹 서버를 라이브러리 형태로 애플리케이션 코드에 포함

D
WAS의 설정 파일을 수동으로 편집



해설
내장 웹 서버는 톰캣 등 웹 서버를 애플리케이션의 라이브러리로 포함시켜 별도 설치 없이 자바 main 메소드 실행만으로 서버를 구동하는 방식입니다.
3. 수동으로 내장 톰캣에 스프링을 통합할 때, 클라이언트 요청을 받아 스프링 컨테이너로 연결하는 핵심 서블릿은 무엇일까요?

A
HelloServlet

B
ContextLoaderListener

DispatcherServlet

D
Connector



해설
DispatcherServlet은 스프링 MVC의 Front Controller로, 내장 톰캣이 받은 요청을 받아 스프링 컨테이너에 등록된 컨트롤러로 라우팅하는 역할을 합니다.
4. 표준 Java JAR 파일이 WAR 파일과 비교하여 라이브러리 포함에 있어 가지는 기본적인 한계는 무엇일까요?

A
실행 권한 설정이 어려움

B
소스 코드를 포함할 수 없음

다른 JAR 파일을 내부에 포함할 수 없음

D
Manifest 파일을 생성할 수 없음



해설
Java 표준 JAR 명세는 JAR 파일 안에 다른 라이브러리 JAR 파일을 직접 포함하는 구조를 지원하지 않습니다. 반면 WAR는 WEB-INF/lib에 가능했죠.
5. 라이브러리 포함 문제를 해결하기 위해 모든 의존성 클래스를 압축 해제하여 하나의 JAR로 묶는 방식을 무엇이라고 할까요?

A
Modular JAR

B
Thin JAR

Fat JAR (또는 Uber JAR)

D
Exploded WAR



해설
Fat JAR는 필요한 모든 라이브러리의 클래스 파일들을 주 JAR 파일 내부로 가져와 하나의 거대한 실행 가능한 JAR로 만드는 방식입니다.
6. Fat JAR 방식의 주요 단점 중 하나는 무엇일까요?

A
빌드 시간이 매우 짧음

B
파일 크기가 항상 작음

C
라이브러리 추적이 용이함

파일 이름 또는 리소스 충돌 발생 가능



해설
Fat JAR는 여러 라이브러리에서 동일한 이름의 클래스나 리소스 파일이 있을 경우, 하나의 파일만 포함되어 충돌이 발생하거나 예상치 못한 동작을 할 수 있습니다.
7. 스프링 부트 애플리케이션을 실행할 때, 임베디드 서버 및 스프링 컨테이너 설정을 포함한 초기화 과정을 자동화하는 핵심 메소드는 무엇일까요?

A
init()

B
startServer()

C
runApplication()

SpringApplication.run()



해설
`SpringApplication.run()` 메소드는 스프링 부트 애플리케이션의 시작점이며, 내장 웹 서버 구동, 스프링 컨테이너 생성 등 복잡한 초기화 과정을 내부적으로 처리합니다.
8. 스프링 부트의 실행 가능 JAR 파일 구조가 일반적인 JAR나 Fat JAR와 다른 독특한 특징은 무엇일까요?

A
라이브러리가 모두 압축 해제되어 클래스만 포함됨

B
WAS 설치가 필수적임

JAR 파일 안에 다른 라이브러리 JAR 파일들을 그대로 포함함

D
매니페스트(Manifest) 파일이 없음



해설
스프링 부트 실행 가능 JAR는 BOOT-INF/lib 등의 폴더 안에 의존성 라이브러리 JAR 파일들을 압축 해제하지 않고 그대로 포함하는 독특한 구조를 가집니다.
9. 스프링 부트의 실행 가능 JAR가 Fat JAR보다 가지는 장점 중 하나는 무엇일까요?

A
파일 크기가 항상 작음

라이브러리 추적 및 식별이 용이함

C
Java 표준 JAR 명세를 완벽히 따름

D
빌드 속도가 매우 빠름



해설
스프링 부트 JAR는 라이브러리 JAR들이 독립적으로 포함되어 있어 Fat JAR와 달리 어떤 라이브러리가 사용되었는지 쉽게 파악할 수 있습니다.
10. `java -jar` 명령으로 스프링 부트 실행 가능 JAR를 실행할 때, Manifest 파일에 명시되어 가장 먼저 실행되는 클래스는 무엇일까요?

A
실제 애플리케이션의 main 메소드

B
Tomcat 시작 클래스

org.springframework.boot.loader.JarLauncher

D
DispatcherServlet



해설
스프링 부트 실행 가능 JAR의 Manifest 파일은 `JarLauncher`를 Main-Class로 지정하여, 이 클래스가 먼저 실행되어 내부 라이브러리를 로드하고 실제 앱의 main 메소드를 호출합니다.
